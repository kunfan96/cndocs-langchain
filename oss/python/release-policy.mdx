本页面说明 LangChain 和 LangGraph 的发布政策。点击下方选项卡查看各自的发布政策：

<Tabs>
    <Tab title="LangChain">

    LangChain 生态系统由不同的组件包组成（例如 `langchain-core`、`langchain`、`langchain-community`、合作伙伴包等）。



    ## 发布节奏

    随着 LangChain 1.0 的发布，`langchain` 和 `langchain-core` 的**次版本**更新（例如从 `1.0.x` 到 `1.1.0`）遵循语义化版本规范，可能会频繁发布。次版本更新包含新功能和改进，但不包含破坏性变更。



    补丁版本发布频繁，每周可能多次，因为它们包含错误修复和小改进。

    ## API 稳定性

    LLM 应用开发是一个快速发展的领域，我们不断从用户和社区中学习。因此，我们预计 `langchain` 和 `langchain-core` 中的 API 将继续演进，以更好地满足用户需求。



    随着 LangChain 1.0 采用语义化版本：
    - 公共 API 的破坏性变更仅在主版本更新中发生（例如 `2.0.0`）
    - 次版本更新（例如 `1.0.0` 到 `1.1.0`）添加新功能，不包含破坏性变更
    - 补丁版本更新（例如 `1.0.0` 到 `1.0.1`）包含错误修复和小改进

    我们通常会尽量避免不必要的变更，并为即将移除的功能提供弃用政策。

    ### 其他包的稳定性

    LangChain 生态系统中其他包的稳定性可能有所不同：

    - **LangChain 维护的合作伙伴包**（如 `langchain-openai` 和 `langchain-anthropic`）遵循语义化版本，预计在 1.0 之后保持稳定。其他合作伙伴包可能遵循不同的稳定性和版本政策，用户应参考这些包的文档了解更多信息。

    - **`langchain-community`** 是一个社区维护的包，包含第三方集成。由于集成数量众多，`langchain-community` 不遵循与 `langchain` 和 `langchain-core` 相同的严格语义化版本政策。更多详情请参阅长期支持下的"特殊考虑"部分。



    ## 弃用政策

    我们通常会在有更好的替代方案之前避免弃用功能。

    根据 LangChain 1.0 的语义化版本方法，已弃用的功能将在整个 1.x 发布系列中继续工作。破坏性变更（包括移除已弃用功能）仅在主版本更新中发生（例如 2.0）。

    当 `langchain` 或 `langchain-core` 中的功能被弃用时，我们将：
    - 在代码和文档中明确标记为已弃用
    - 提供迁移到推荐替代方案的指南
    - 在所有 1.x 次版本中为已弃用功能提供安全更新



    在某些情况下，如果已弃用的功能不会造成维护问题，我们可能会允许其在代码库中保留更长时间，以进一步减轻用户的负担。

    ## 长期支持 (LTS)

    LangChain 遵循长期支持 (LTS) 政策，为生产应用提供稳定性：

    ### 发布状态定义

    包被标记为以下状态之一：
    - **ACTIVE**：当前活跃开发，包括错误修复、安全补丁和新功能
    - **MAINTENANCE**：接收所有安全补丁和关键错误修复，但不包含新功能

    ### 当前 LTS 版本

    **LangChain 1.0** 被指定为 LTS 版本：
    - **状态**：ACTIVE，直到 2.0 发布
    - **支持期限**：2.0 发布后，1.0 将进入 MAINTENANCE 模式至少 1 年
    - **语义化版本合规**：用户可以在次版本之间升级（例如 1.0 到 1.1），不会有破坏性变更

    ### 旧版本支持

    **LangChain 0.3**：
    - **状态**：MAINTENANCE 模式
    - **支持期限**：至 2026 年 12 月
    - **支持包括**：安全补丁和关键错误修复

    ### 特殊考虑

    **langchain-community 0.4**：由于社区贡献和第三方集成的性质，`langchain-community` 可能在次版本更新中有破坏性变更。它已发布为 0.4 版本以反映这种不同的稳定性政策。



    </Tab>
    <Tab title="LangGraph">
    LangGraph 遵循结构化的发布政策，以确保构建生产应用的用户获得稳定性和可预测性。

    ## 发布节奏

    我们预计**主版本**更新间隔至少 6-12 个月，以为生产应用提供稳定性。

    **次版本**更新通常每 1-2 个月发布一次，包含新功能和改进。

    **补丁**版本根据需要发布，通常每周发布，以解决错误和安全问题。

    ## API 稳定性

    ### 稳定 API

    所有没有特殊前缀的 API 都被视为稳定的，可用于生产环境。我们在主版本内为稳定功能保持向后兼容性。

    ### Beta 功能

    文档中标记为 `beta` 的功能：

    - 功能完整且经过测试
    - 可以安全地用于生产，但需了解它们可能会更改
    - 可能会根据用户反馈进行小幅 API 调整

    ### 实验性功能

    标记为 `experimental` 或 `alpha` 的功能：

    - 正在积极开发中
    - 可能会发生重大变更或被移除
    - 在生产环境中应谨慎使用

    ### 内部 API

    以下划线 (`_`) 为前缀或明确标记为内部的 API：

    - 不是公共 API 的一部分
    - 可能会在没有通知的情况下更改
    - 不应直接使用

    ## 弃用政策

    弃用功能时：

    1. **弃用通知**：功能被标记为已弃用，并附有明确的迁移指南
    2. **宽限期**：已弃用的功能至少在一个次版本内保持功能正常
    3. **移除**：功能仅在主版本更新中被移除
    4. **迁移支持**：我们提供迁移指南，并在可能的情况下提供自动化工具

    ## 平台兼容性

    ### Python 支持

    - 我们支持 Python 软件基金会积极维护的 Python 版本
    - Python 版本要求仅在主版本更新中更改
    - 目前需要 Python 3.10 或更高版本




    ## 破坏性变更

    破坏性变更仅在主版本中引入，包括：
    - 移除已弃用的 API
    - 更改必需参数
    - 影响现有应用的默认行为更改
    - Python/Node.js 最低版本更新

    ## 迁移支持

    对于主版本升级，我们提供：
    - 全面的迁移指南
    - 在可行的情况下提供自动迁移脚本
    - 为上一个主版本提供延长支持期
    - 所有破坏性变更的清晰文档

    ## 长期支持 (LTS)

    LangGraph 遵循长期支持 (LTS) 政策，为生产应用提供稳定性：

    ### 发布状态定义

    包被标记为以下状态之一：
    - **ACTIVE**：当前活跃开发，包括错误修复、安全补丁和新功能
    - **MAINTENANCE**：接收所有安全补丁和关键错误修复，但不包含新功能

    ### 当前 LTS 版本

    **LangGraph 1.0** 被指定为 LTS 版本：
    - **状态**：ACTIVE，直到 2.0 发布
    - **支持期限**：2.0 发布后，1.0 将进入 MAINTENANCE 模式至少 1 年
    - **语义化版本合规**：用户可以在次版本之间升级（例如 1.0 到 1.1），不会有破坏性变更

    ### 旧版本支持

    **LangGraph 0.4**：
    - **状态**：MAINTENANCE 模式
    - **支持期限**：至 2026 年 12 月
    - **支持包括**：所有安全补丁和关键错误修复
