本页解释了 LangChain 和 LangGraph 的发布政策。点击下方标签页查看各自的发布政策：

<Tabs>
    <Tab title="LangChain">


    LangChain 生态系统由不同的组件包组成（例如 `@langchain/core`、`langchain`、`@langchain/community`、合作伙伴包等）。


    ## 发布节奏


    随着 LangChain 1.0 的发布，`langchain` 和 `@langchain/core` 的**次要**版本（例如从 `1.0.x` 到 `1.1.0`）遵循语义化版本控制，并且可能频繁发布。次要版本包含新功能和改进，但不包含破坏性更改。


    补丁版本会频繁发布，每周最多几次，因为它们包含错误修复和次要改进。

    ## API 稳定性


    LLM 应用程序的开发是一个快速发展的领域，我们不断从用户和社区中学习。因此，我们预计 `langchain` 和 `@langchain/core` 中的 API 将继续发展，以更好地满足用户的需求。


    随着 LangChain 1.0 采用语义化版本控制：

    - 公共 API 的破坏性更改仅在主要版本发布时发生（例如 `2.0.0`）
    - 次要版本升级（例如 `1.0.0` 到 `1.1.0`）会添加新功能而不包含破坏性更改
    - 补丁版本升级（例如 `1.0.0` 到 `1.0.1`）包含错误修复和次要改进

    我们通常会尽量避免不必要的更改，并为被移除的功能提供弃用策略。

    ### 其他包的稳定性

    LangChain 生态系统中其他包的稳定性可能有所不同：

    - **由 LangChain 维护的合作伙伴包**（例如 `langchain-openai` 和 `langchain-anthropic`）遵循语义化版本控制，并预计在 1.0 后保持稳定。其他合作伙伴包可能遵循不同的稳定性和版本控制策略，用户应参考这些包的文档以获取更多信息。


    - **`@langchain/community`** 是一个社区维护的包，包含第三方集成。由于社区贡献和第三方依赖的性质，`@langchain/community` 不遵循与 `langchain` 和 `@langchain/core` 相同的严格语义化版本控制策略。有关更多详细信息，请参见长期支持下的“特殊考虑”部分。


    ## 弃用策略

    我们通常会避免弃用功能，直到有更好的替代方案可用。


    随着 LangChain 1.0 的语义化版本控制方法，弃用的功能将在整个 1.x 发布系列中继续工作。破坏性更改，包括移除弃用的功能，仅在主要版本发布时发生（例如 2.0）。

    当 `langchain` 或 `@langchain/core` 中的功能被弃用时，我们将：
    - 在代码和文档中明确标记为已弃用
    - 提供迁移到推荐替代方案的指导
    - 通过所有 1.x 次要版本为弃用功能提供安全更新


    在某些情况下，如果弃用功能不会导致维护问题，我们可能会允许它们在代码库中保留更长时间，以进一步减轻用户的负担。

    ## 长期支持 (LTS)

    LangChain 遵循长期支持 (LTS) 政策，为生产应用程序提供稳定性：

    ### 发布状态定义

    包被标记为以下状态之一：
    - **ACTIVE**：当前活跃开发，包括错误修复、安全补丁和新功能
    - **MAINTENANCE**：接收所有安全补丁和关键错误修复，但不包含新功能

    ### 当前 LTS 发布

    **LangChain 1.0** 被指定为 LTS 发布：
    - **状态**：ACTIVE，直到 2.0 发布
    - **支持周期**：2.0 发布后，1.0 将进入 MAINTENANCE 模式至少 1 年
    - **语义化版本控制合规性**：用户可以在次要版本之间升级（例如 1.0 到 1.1）而不会出现破坏性更改

    ### 旧版本支持

    **LangChain 0.3**：

    - **状态**：MAINTENANCE 模式
    - **支持周期**：直到 2026 年 12 月
    - **支持内容**：安全补丁和关键错误修复

    ### 特殊考虑


    **@langchain/community 1.0**：由于社区贡献和第三方集成的性质，`@langchain/community` 可能在次要版本中出现破坏性更改。

    <Tab title="LangGraph">
    LangGraph 遵循结构化的发布政策，以确保为构建生产应用程序的用户提供稳定性和可预测性。

    ## 发布节奏

    我们预计**主要**版本之间至少间隔 6-12 个月，以提供生产应用程序的稳定性。

    **次要**版本通常每 1-2 个月发布一次，包含新功能和改进。

    **补丁**版本根据需要发布，通常每周一次，以解决错误和安全问题。

    ## API 稳定性

    ### 稳定 API

    所有没有特殊前缀的 API 都被视为稳定并准备好用于生产。我们在主要版本内维护稳定功能的向后兼容性。

    ### Beta 功能

    文档中标记为 `beta` 的功能：

    - 功能完整且经过测试
    - 可安全用于生产，但需理解它们可能会更改
    - 可能根据用户反馈进行次要 API 调整

    ### 实验性功能

    标记为 `experimental` 或 `alpha` 的功能：

    - 正在积极开发中
    - 可能发生重大更改或被移除
    - 在生产中使用时应谨慎

    ### 内部 API

    以 underscore (`_`) 为前缀或明确标记为内部的 API：

    - 不是公共 API 的一部分
    - 可能在不通知的情况下更改
    - 不应直接使用

    ## 弃用策略

    弃用功能时：

    1. **弃用通知**：功能被标记为已弃用，并提供清晰的迁移指导
    2. **宽限期**：弃用功能至少在一个次要版本中保持功能
    3. **移除**：功能仅在主要版本发布时移除
    4. **迁移支持**：我们提供迁移指南，并在可能时提供自动化工具

    ## 平台兼容性



    ### JavaScript/TypeScript 支持

    - 我们支持 Node.js LTS 版本
    - 为所有公共 API 提供 TypeScript 定义
    - 客户端组件的浏览器兼容性有文档记录


    ## 破坏性更改

    破坏性更改仅在主要版本中引入，包括：

    - 移除已弃用的 API
    - 更改必需参数
    - 更改影响现有应用程序的默认行为
    - 最低 Python/Node.js 版本更新

    ## 迁移支持

    对于主要版本升级，我们提供：

    - 全面的迁移指南
    - 可行时的自动化迁移脚本
    - 旧主要版本的扩展支持期
    - 所有破坏性更改的清晰文档

    ## 长期支持 (LTS)

    LangGraph 遵循长期支持 (LTS) 政策，为生产应用程序提供稳定性：

    ### 发布状态定义

    包被标记为以下状态之一：

    - **ACTIVE**：当前活跃开发，包括错误修复、安全补丁和新功能
    - **MAINTENANCE**：接收所有安全补丁和关键错误修复，但不包含新功能

    ### 当前 LTS 发布

    **LangGraph 1.0** 被指定为 LTS 发布：

    - **状态**：ACTIVE，直到 2.0 发布
    - **支持周期**：2.0 发布后，1.0 将进入 MAINTENANCE 模式至少 1 年
    - **语义化版本控制合规性**：用户可以在次要版本之间升级（例如 1.0 到 1.1）而不会出现破坏性更改

    ### 旧版本支持

    **LangGraph 0.4**：

    - **状态**：MAINTENANCE 模式
    - **支持周期**：直到 2026 年 12 月
    - **支持内容**：所有安全补丁和关键错误修复
